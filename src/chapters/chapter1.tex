\section{Stating the problem}
At the start of this internship, the company's entire application stack---comprising two frontend applications (Vue.js and React) and one backend service (NestJS)---was deployed on a single Amazon EC2 instance. This instance served as the sole runtime environment for all services, with no separation between components and no redundancy in place. Deployments were performed manually by connecting to the instance via SSH, pulling the latest code, and restarting the services by hand.

This approach, while sufficient during the earliest stages of development, introduced several critical limitations as the company prepared to move toward a production-ready product. The most pressing issue was the complete absence of scalability: a single virtual machine cannot dynamically allocate resources in response to varying traffic loads. During usage spikes, performance degradation was inevitable, and during idle periods, the fixed instance cost remained unchanged regardless of demand.

Beyond scalability, the automation in place was minimal. A basic CI/CD pipeline existed that built Docker images and pushed them to Amazon ECR, but the process ended there. The actual deployment---pulling images onto the EC2 instance, stopping running containers, and restarting services---was still performed manually via SSH. There was no continuous deployment, no automated rollback mechanism, and no integration between the image registry and the runtime environment. The absence of Infrastructure as Code meant that the server configuration existed only in the state of the running machine---undocumented, unreproducible, and vulnerable to configuration drift.

Furthermore, the architecture offered no high availability or fault tolerance. If the EC2 instance failed or needed maintenance, all services went offline simultaneously. There was no load balancing, no health checking, and no mechanism for automatic recovery. The database ran alongside the application services on the same instance, creating a single point of failure for both compute and data.

This situation made it clear that a fundamental redesign of the infrastructure was necessary. The company needed a modern, cloud-native architecture capable of scaling with demand, automating deployments, isolating environments, and providing the reliability expected of a production system.

\section{Assessment of the case}
\subsection{Describing the work procedure}
The work on any project must first of all be preceded by a thorough study of the existing ones which undermines the strengths and weaknesses of the current system, as well as the business decisions that should be taken into account during the conception as well as the realization.

\subsection{Criticizing the current state}
A thorough assessment of the existing infrastructure revealed several structural deficiencies that made it unsuitable for production use. These issues can be categorized as follows:

\begin{itemize}
    \item \textbf{Single point of failure:} All application components---frontends, backend, and database---resided on one EC2 instance. Any hardware failure, network issue, or misconfiguration would result in a complete service outage with no automatic failover mechanism.

    \item \textbf{No scalability:} The infrastructure had no ability to scale horizontally. Adding capacity required manually provisioning new instances and reconfiguring the application, a process that could not respond to real-time demand fluctuations.

    \item \textbf{Manual deployments:} Each deployment involved SSH access to the server, manual code pulls, dependency installation, and service restarts. This process was slow, inconsistent, and prone to human error, with no rollback mechanism in case of failure.

    \item \textbf{No environment separation:} There was no distinction between staging and production environments. Testing changes meant deploying directly to the live system, increasing the risk of introducing bugs or downtime for end users.

    \item \textbf{Unreproducible infrastructure:} The server was configured manually over time, with no record of the exact steps taken. Recreating the same environment on a new machine would require reverse-engineering the existing setup, a fragile and unreliable process.

    \item \textbf{No monitoring or observability:} There were no health checks, alerting systems, or logging aggregation in place. Issues could go undetected until users reported them, and diagnosing root causes required manual investigation on the server.
\end{itemize}

These shortcomings collectively represented a significant technical debt that would only grow as the application matured and the user base expanded. Addressing them required not incremental improvements, but a complete rethinking of how the infrastructure was designed, provisioned, and managed.

\subsection{Proposed solution}
To address the limitations identified above, we proposed building a complete cloud-native infrastructure from scratch on Amazon Web Services (AWS). The solution encompasses every layer of the stack, from network design to automated deployment, and is guided by modern DevOps best practices.

\subsubsection*{\underline{Container orchestration with Kubernetes}}
The application stack is migrated from a single EC2 instance to Amazon Elastic Kubernetes Service (EKS). Kubernetes provides automatic container scheduling, self-healing capabilities, horizontal pod autoscaling, and rolling updates with zero downtime. The cluster is designed with a multi-environment architecture: a production namespace hosting all three applications (two frontends and one backend) and a staging namespace for pre-production testing.

\subsubsection*{\underline{Infrastructure as Code with Terraform}}
All AWS resources---VPC, subnets, security groups, EKS cluster, RDS databases, S3 buckets, NAT and Internet Gateways, and the Application Load Balancer (ALB)---are defined and provisioned using Terraform. This ensures that the entire infrastructure is version-controlled, reproducible, and can be torn down and recreated consistently across environments.

\subsubsection*{\underline{Containerization with Docker}}
Each application (Vue.js, React, NestJS) is containerized using Docker, producing lightweight and portable images. These images are stored in GitHub Container Registry (GHCR), chosen over AWS ECR as a cost optimization measure. Containerization guarantees consistency between development, staging, and production environments.

\subsubsection*{\underline{CI/CD automation with GitHub Actions}}
Continuous integration and continuous delivery pipelines are implemented using GitHub Actions. On each push to the repository, the pipeline automatically builds the application, runs tests, constructs a Docker image, pushes it to GHCR, and updates the deployment configuration in the GitOps repository. This eliminates all manual deployment steps.

\subsubsection*{\underline{GitOps with ArgoCD and Kustomize}}
Continuous deployment is handled through a GitOps approach using ArgoCD. A dedicated GitOps repository contains all Kubernetes manifests, organized with Kustomize overlays for staging and production environments. ArgoCD monitors this repository and automatically synchronizes the cluster state with the declared configuration in Git, ensuring that the deployed state always matches what is committed in version control.

\subsubsection*{\underline{Local testing with Minikube}}
Before deploying to the cloud, all Kubernetes manifests and Kustomize overlays are validated locally using Minikube. This local Kubernetes cluster allows testing deployments, service connectivity, and configuration correctness without incurring cloud costs or risking production stability.

\section{Development Methodology}
\subsection{Agile methodology}
Agile is a structured and iterative approach to project management and product development.
It recognizes the volatility of product development, and provides a methodology for self-organizing teams to respond to change without going off the rails.

\subsection{Scrum methodology}
Scrum teams commit to completing an increment of work, which is potentially shippable, through set intervals called sprints.
Their goal is to create learning loops to quickly gather and integrate customer feedback.
Scrum teams adopt specific roles, create special artifacts, and hold regular ceremonies to keep things moving forward.

\subsection{Kanban methodology}
Kanban is all about visualizing your work, limiting work in progress, and maximizing efficiency (or flow).
Kanban teams focus on reducing the time a project takes from start by using a Kanban board to continuously improve their flow of work.
To explain more in details, Kanban is based on a continuous workflow structure that keeps teams nimble and ready to adapt to changing priorities.
Work items ---represented by cards--- are organized on the board where they flow from one stage of the workflow or column to the next.
Common workflow stages are To Do, In Progress, In Review, and Done.

\subsection{The choice for our project}
Infrastructure work is inherently unpredictable---a networking issue can block an entire week's plan, while a Terraform module might be completed in hours instead of days. Unlike application development where feature scope is relatively well-defined, DevOps tasks frequently shift in priority based on blockers, dependencies, and discoveries made during implementation.

For this reason, Kanban was chosen over Scrum as our development methodology. Its continuous flow model allowed us to reprioritize tasks dynamically without the overhead of sprint planning ceremonies. Combined with Asana as our project management tool, we maintained a Kanban board where each infrastructure component---Terraform modules, Kubernetes configuration, CI/CD pipelines, ArgoCD setup---was tracked as a card moving through To Do, In Progress, In Review, and Done stages.

This setup provided full visibility into the project's progress for all team members, while giving the flexibility needed to adapt to the evolving requirements of building cloud infrastructure from scratch.

\section{Conclusion}
In this chapter, we presented the host organization and the general context of the internship. We then identified the limitations of the existing infrastructure---a single EC2 instance with minimal automation---and proposed a comprehensive cloud-native solution built on AWS, Kubernetes, Terraform, and GitOps practices. Finally, we discussed the development methodology adopted for this project, choosing Kanban for its flexibility in managing the inherently unpredictable nature of infrastructure work.
