\section{General framework of the internship}
This project was carried out within the frame of obtaining a bachelor's degree in Computer Science at the Faculty of Science of Monastir (FSM), Tunisia.
The internship took place over 4 months at The Mashroom, an innovative early-stage startup focused on 3D product visualization services.
This section introduces the host company {\bf The Mashroom} as well as the services it offers.

visuals on demand. The company significantly reduces costs and production time compared to traditional photography shoots.

\section{Stating the problem}
At the beginning of the internship, the company's entire software platform (composed of 2 frontend applications, a backend application, and a database)
was running on a \textbf{single server} hosted on a cloud provider, which rents servers and compute resources. Examples of cloud providers include Amazon Web Services \cite{aws} and Google Cloud \cite{gcp}.
This single-machine setup introduced several limitations. There was no separation between components since they all shared the same resources, so if one of them had a new release the whole system would go down with it. A system administrator would have to remotely connect to the server, pull the latest code, and manually restart the affected services (frontend, backend, or database), which left room for human error at every step.

This setup was sufficient during the early stages of development, but as the company was preparing to launch its services to real customers it could no longer handle more users due to limited machine resources. It was no longer reliable or flexible enough to support the upcoming release.

\section{Assessment of the case}
Before stepping into designing a new infrastructure, this section presents the current 
system used by The Mashroom, the solutions commonly used in the industry for similar problems,
and concludes with the architectural direction adopted for this project.

\subsection{Criticizing the current state}
The Mashroom's platform is composed of several components, including frontend applications, 
a backend service, and a database, each developed and deployed onto the same infrastructure. 
In this setup, every change made to any component goes through a manual process: a developer 
connects to the server, pulls the latest code, installs dependencies, and restarts the service. 
This section analyzes the limitations of this current state and highlights the main issues 
that need to be addressed.
\begin{enumerate}
    \item \textbf{Single point of failure:} every component of The Mashroom platform, 
from the web interfaces (2 frontend applications) to the backend services and 
the database, runs on the same machine. Any hardware failure, network issue, 
or misconfiguration would take the entire platform offline at once, with no 
automatic recovery in place.

    \item \textbf{No scalability:} the platform cannot handle more users without manual 
intervention. Increasing capacity means provisioning a new server by hand and 
reconfiguring the application to ensure it runs correctly on the new infrastructure. This process is far too slow to 
react to real-time traffic spikes.

    \item \textbf{Manual deployments:} each release was carried out manually by a 
developer connected to the server, who pulled the new code, installed its 
dependencies, and restarted the running service. This process was slow, 
inconsistent, error-prone, and offered no mechanism to revert a faulty release.

    \item \textbf{No environment separation:} there was no dedicated environment in which to try a change before exposing it to real users. 
    Every modification was applied directly to the live service, 
    so any bug introduced during development was felt immediately by users.

    \item \textbf{Unreproducible infrastructure:} the server had been configured by hand over 
The Mashroom's development period with no record of the exact steps taken. 
Rebuilding the same environment on a new machine would require reconstructing 
the configuration piece by piece from scratch.

    \item \textbf{High operational cost:} maintaining the current setup required 
significant time from developers to handle deployments, debug infrastructure 
issues, and manage the server manually. Combined with the cost of renting a 
machine that could not scale to meet demand, the setup was neither time 
efficient nor cost effective.
\end{enumerate}

These limitations directly shape the requirements that the new architecture 
must satisfy.


\subsection{Study of existing solutions}
This section reviews three widely used architectural and deployment approaches for modern web applications.

\subsubsection*{\underline{Function-as-a-Service execution (FaaS) \cite{faas}}}
The first approach is to run the application logic as short-lived functions 
that are triggered by an event and executed on demand. AWS Lambda and Google 
Cloud Functions are examples of this service offered by cloud providers. This 
model is attractive for occasional workflows but it is not suited to serve 
platforms like The Mashroom, since it is not meant to run for long periods of time 
due to the execution time limits these services enforce. Additionally, it can 
become costly for continuously running workloads, as pricing is based on 
execution time and the number of requests.

\subsubsection*{\underline{Fully managed container services}}
A second approach relies on third-party services that accept a packaged container 
and run it without requiring the developer to manage the underlying infrastructure 
or servers. This approach was popularized in the early 2010s with platforms like 
Heroku and Google App Engine \cite{heroku2011}, and its main advantage is convenience.
However, this convenience comes with tradeoffs: these platforms charge a premium 
for handling the infrastructure, and they offer little control over how the system 
is configured and operated. For a platform like The Mashroom, this could mean higher 
costs over time and a lack of flexibility when specific infrastructure decisions 
are needed.

\subsubsection*{\underline{Managed container orchestration}}
A third approach is to run the application on a coordinated group of machines 
called a cluster, managed by a piece of software called an orchestrator. Every 
major cloud provider now offers this as a managed service, examples of which 
are Amazon's EKS \cite{aws_eks}, Google's GKE \cite{google_gke}, and 
Microsoft's AKS \cite{microsoft_aks}. This removes the burden of operating 
the orchestration layer but comes with an extra management charge. For a small 
company without a dedicated infrastructure team, this option can quickly become 
more expensive than the others.

\section{Proposed solution}
The limitations identified in the previous section are divided into three 
complementary layers, each addressing a distinct set of the six weaknesses 
discussed above. Together they directly resolve the identified shortcomings 
and prepare The Mashroom for a smooth production launch.

\subsubsection*{\underline{Container Orchestration Migration}}
The application is migrated off its single virtual machine onto a managed 
container orchestration cluster. Each component of The Mashroom platform, 
the frontend applicatio ns, the backend services, and the database, is 
packaged so that it can run consistently on any machine in the cluster. 
The orchestrator handles the management of these components, ensuring they 
remain available, recover from failure, and scale with demand. Both the 
production and pre-production environments are hosted on the same cluster, 
kept isolated from each other using the cluster's isolation mechanisms, 
avoiding the additional cost and complexity of maintaining a separate cluster 
for each environment.

\subsubsection*{\underline{Reproducible cloud infrastructure}}
The entire infrastructure is defined as code \cite{iac-definition}, meaning every component of the infrastructure is
described in versioned files rather than configured by hand. This guarantees 
that the infrastructure can be reproduced identically on any environment at 
any time and that every change is tracked and auditable. Even sensitive 
application credentials are handled in a structured and automated way, 
removing any need for manual intervention when managing sensitive 
configuration values.

\subsubsection*{\underline{Automated and reversible delivery}}
The third layer addresses how changes reach the cluster. When a developer 
pushes new code, an automated sequence of steps is triggered to build, test,
and package the application into a deployable unit. This sequence is called
a pipeline \cite{cicd-definition}. Once the pipeline completes successfully, the changes are forwarded 
to the cluster, which picks them up and applies the new version to the 
appropriate environment. If a hidden problem surfaces after a release, the 
faulty change can be undone and the application is restored to its last stable 
version before the issue affects the majority of users. This makes the delivery 
process fully automated yet fully reversible.

\section{Development Approach}
In order to develop the proposed solution based on its basic objetives, we adopt the DevOps approach. DevOps is a set of practices that brings together software development and IT
operations with the goal of shortening the delivery cycle and increasing
reliability through automation. According to Microsoft Azure,
the DevOps lifecycle operates as a continuous loop spanning four primary phases:
Plan, Develop, Deliver, and Operate \cite{devops}.

\begin{figure}[H]
    \centering
    \makebox[\textwidth]{\includegraphics[width=12cm]{src/assets/images/devops-lifecycle.png}}
    \caption{DevOps and the application lifecycle}
    \label{fig:devops-lifecycle}
\end{figure}

In this project, these phases map directly to the work carried out:
\begin{itemize}
    \item \textbf{Develop (Continuous Integration):} Replacing manual code testing with Continuous Integration (CI) \cite{cicd-definition} pipelines that autonomously validate code across the various frontend and backend repositories, packaging them into standardized container images.
    \item \textbf{Deliver (Continuous Deployment):} Transitioning from manual server updates to automated deployment mechanisms. Managing multiple distributed services forces a critical architectural decision between traditional push-based deployments and modern pull-based "GitOps" \cite{gitops-definition} methodologies.
    \item \textbf{Operate (Infrastructure as Code):} Eliminating the manual configuration of cloud resources by adopting Infrastructure as Code (IaC) \cite{iac-definition}, ensuring the new distributed environment is reproducible, auditable, and dynamically scalable.
\end{itemize}

DevOps work is rarely linear: tasks shift in priority, and blockers can appear
at any stage. Kanban \cite{kanban} suits this well, since it focuses on continuous flow rather
than fixed sprints, keeping work visible and moving at all times. Throughout the
internship, all tasks were tracked on a Kanban board using Asana.
