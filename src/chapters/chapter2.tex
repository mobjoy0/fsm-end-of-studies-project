\section{Introduction}
This chapter examines the core concepts and technologies that underpin our
internship project: DevOps culture, containerization, infrastructure as code,
CI/CD pipelines, and cloud-native deployment strategies. We begin by defining
these foundational practices and then conduct a comparative analysis of the
tools available in each domain, from version control systems to container
orchestration platforms and cloud providers.

The objective is twofold: first, to establish a clear theoretical grounding
for the reader; and second, to justify the technical choices made throughout
this project by evaluating each option against the specific requirements of
migrating a monolithic application to a scalable, cloud-based infrastructure.

\section{The DevOps Paradigm}

\subsection{Definition}

DevOps represents a fundamental cultural and technical shift designed to accelerate the Software Development Lifecycle (SDLC) by bridging the gap between software engineering and IT operations. While development focuses on iterating and improving the application layer, operations manage the provisioning, security, and scalability of the underlying infrastructure.

Prior to this project, the startup's existing footprint consisted of a simple website hosted on a single Amazon EC2 (Elastic Compute Cloud) instance. However, the core product under development is a complex, distributed system comprising multiple frontends (Vue.js and React), a robust backend (NestJS), and resource-intensive 3D AI generative models. A single-VM infrastructure is entirely insufficient for this architecture; it lacks isolated scaling, introduces a single point of failure, and requires manual deployment interventions.

Therefore, applying DevOps practices in this context is not merely an optimization, but a foundational necessity. The objective is to architect a highly scalable, automated, cloud-native infrastructure from scratch that can support a distributed application suite.

\subsection{Lifecycle}
\begin{figure}[H]
    \centering
    \makebox[\textwidth]{\includegraphics[width=12cm]{src/assets/images/devops-lifecycle.png}}
    \caption{DevOps and the application lifecycle}
    \label{fig:devops-lifecycle}
\end{figure}

According to Microsoft Azure, the DevOps lifecycle operates as a continuous loop spanning four primary phases: Plan, Develop, Deliver, and Operate. To successfully build and maintain this new multi-component system, these phases must rely heavily on extreme automation:
\begin{itemize}
    \item \textbf{Develop (Continuous Integration):} Replacing manual code testing with Continuous Integration (CI) pipelines that autonomously validate code across the various frontend and backend repositories, packaging them into standardized container images.
    \item \textbf{Deliver (Continuous Deployment):} Transitioning from manual server updates to automated deployment mechanisms. Managing multiple distributed services forces a critical architectural decision between traditional push-based deployments and modern pull-based "GitOps" methodologies.
    \item \textbf{Operate (Infrastructure as Code):} Eliminating the manual configuration of cloud resources by adopting Infrastructure as Code (IaC), ensuring the new distributed environment is reproducible, auditable, and dynamically scalable.
\end{itemize}

\subsection{Container management}
Containers have become an integral part of DevOps over the past couple of years but what are containers exactly and how do we manage them?

\subsubsection*{What is a container ?}
Simply said; container is a packaging format for software applications that are akin to a very lightweight virtual machine (very broad analogy) which always executes in an isolated environment \cite{what-is-a-container}.
What this implies is that said containers can be easily copied to and run on different machines with high reliability, lower costs and high efficiency.

\subsubsection*{What is container management ?}
Container management is the process for automating the creation, deployment and scaling of containers \cite{container-management}.
Container management tools such as Docker and Podman facilitate the addition, replacement and organization of containers.

\subsection{CI/CD pipelines}
A continuous integration and continuous deployment (CI/CD) pipeline is a series of steps that must be performed in order to deliver a new version of software.
CI/CD pipelines are a practice focused on improving software delivery throughout the software development life cycle via automation. \cite{cicd-definition}

\subsection{Infrastructure as Code}
Infrastructure as Code (IaC) is the practice of managing and provisioning computing infrastructure through machine-readable definition files rather than through manual processes or interactive configuration tools \cite{iac-definition}. Instead of manually configuring servers, networks, and services through a cloud provider's console, engineers write declarative or imperative code that describes the desired state of the infrastructure.

IaC brings software engineering practices to infrastructure management: version control, code review, testing, and reproducibility. This means that an entire cloud environment can be created, modified, or destroyed consistently and repeatedly from the same set of configuration files. In the context of our project, IaC is essential for provisioning the AWS resources that host the application, ensuring that environments can be reliably recreated and that infrastructure changes are auditable and reversible.

\subsection{GitOps}
GitOps is an operational framework that extends Infrastructure as Code principles by using Git as the single source of truth for both infrastructure and application configurations \cite{gitops-definition}. In a GitOps workflow, the desired state of the system is declared in a Git repository, and an automated agent running inside the cluster continuously reconciles the actual state with the declared state.

This approach provides several benefits: every change is versioned and auditable through Git history, rollbacks are as simple as reverting a commit, and the deployment process becomes declarative rather than imperative. GitOps eliminates the need for direct cluster access during deployments, which improves both security and reliability.

\section{Comparative Analysis}
To implement a scalable and automated infrastructure, a range of tools is
required for version control, container management, deployment orchestration,
and cloud provisioning. In this section, we provide a comparative study of
widely used tools and cloud platforms in the industry, highlighting their
main characteristics and advantages. This analysis allows us to identify
which solutions are most suitable for modernizing and automating the
deployment of applications in cloud environments.

When deploying applications to Kubernetes, two dominant approaches exist for managing manifests across multiple environments: Kustomize and Helm. Both solve the problem of environment-specific configuration but take fundamentally different approaches.

        \hline
        \textbf {Advantages}                & \begin{itemize}[leftmargin=*, topsep=0pt, itemsep=1pt, parsep=2pt]

For our project, Kustomize was chosen as the primary configuration management tool. Its overlay-based approach maps naturally to our multi-environment architecture (staging and production namespaces), integrates natively with ArgoCD for GitOps deployments, and keeps manifests readable as plain Kubernetes YAML without introducing templating complexity.

Its modular architecture and the ability to preview changes with \texttt{terraform plan} before applying them provide an additional layer of safety when managing production infrastructure.

\section{Conclusion}
In this chapter, we established the theoretical and technical foundations for our project by examining DevOps practices, containerization, infrastructure as code, CI/CD pipelines, and GitOps principles. We then conducted a comparative analysis of the tools available in each domain: Git for version control, Docker for containerization, Kubernetes for orchestration, Kustomize for configuration management, GitHub Actions for CI/CD, Terraform for infrastructure provisioning, and AWS as the cloud platform.

Each choice was justified against the specific requirements of our project: migrating a monolithic application to a scalable, cloud-native architecture while optimizing for cost, maintainability, and automation. These tools and practices form the technical stack that will be implemented in the subsequent chapters.
